Key idea

Efficiency is a property of the algorithm, not the computer. We measure it by counting steps as a function of input size \(n\).

Growth rates and Big-O

Why counting seconds fails

  • Timing an algorithm depends on the machine, the language, the load, the input…
  • Instead: count basic operations (assignments, comparisons, arithmetic) as a function of input size \(n\).
  • Two algorithms solving the same problem can differ astronomically in operation count.
Same problem, two costs

Sum \(1 + 2 + \dots + n\):

  • Loop and add: \(\approx n\) operations
  • \(\dfrac{n(n+1)}{2}\): \(\approx 3\) operations

Counting operations: an example

Algorithm SumPairs(L):    total ← 0    Foreach x in L Do        Foreach y in L Do            total ← total + x * y    Return total
  • Let \(n\) be the length of L. Line 2: 1 assignment. Lines 3–5: the inner body runs \(n \times n = n^2\) times.
  • Total \(\approx 2n^2 + n + 2\) operations. As \(n\) grows, the \(n^2\) term dominates.
Key idea

For large \(n\), only the fastest-growing term matters — constants and lower-order terms fade into irrelevance.

Big-O: the formal definition

Definition

\(f(n) = O(g(n))\) if and only if there exist constants \(c > 0\) and \(n_0 \geq 0\) such that

\[ f(n) \;\leq\; c \cdot g(n) \qquad \text{for all } n \geq n_0. \]

  • In words: beyond some point \(n_0\), \(f\) is bounded above by a constant multiple of \(g\).
  • Example: \(f(n) = 2n^2 + n + 2\) is \(O(n^2)\). Take \(c = 5\), \(n_0 = 1\): for all \(n \geq 1\), \(2n^2 + n + 2 \leq 5n^2\). ✓
  • Big-O is an upper bound — it says “grows no faster than”.

Show that \(10n + 25\) is \(O(n)\) by finding suitable \(c\) and \(n_0\).

Big-O arithmetic (the working rules)

  1. Drop constants: \(O(3n) = O(n)\);  \(O(n^2/2) = O(n^2)\).
  2. Drop lower-order terms: \(O(n^2 + n + \log n) = O(n^2)\).
  3. Sequential blocks add: \(O(f)\) then \(O(g)\) \(\Rightarrow O(f + g) = O(\max(f,g))\).
  4. Nested loops multiply: a body of \(O(f)\) run \(O(g)\) times \(\Rightarrow O(f \cdot g)\).

\(\Theta\), \(\Omega\), and best / average / worst case

Big-O's siblings: \(\Omega\) and \(\Theta\)

  • \(f(n) = \Omega(g(n))\): \(f\) grows at least as fast as \(g\) (lower bound) — \(f(n) \geq c \cdot g(n)\) for \(n \geq n_0\).
  • \(f(n) = \Theta(g(n))\): \(f\) grows exactly as fast as \(g\) — both \(O(g)\) and \(\Omega(g)\).
Example

\(f(n) = 2n^2 + n\) is: \(O(n^2)\) ✓, \(\Omega(n^2)\) ✓, so \(\Theta(n^2)\).
It is also \(O(n^3)\) (true but unhelpfully loose) and \(\Omega(n)\) (also loose).

Convention

When we say “the algorithm is \(O(n^2)\)” we usually mean the tightest bound we can prove. Loose bounds are technically correct but score poorly.

Best, average, worst — three questions, one algorithm

Algorithm LinearSearch(L, target):    Foreach i from 1 To length(L) Do        If L[i] = target Do            Return i    Return not_found
  • Best case: target is first — \(\Theta(1)\).   Worst case: target is last or absent — \(\Theta(n)\).
  • Average case: target uniformly random in L — \(\approx n/2\) comparisons, \(\Theta(n)\).
Key idea

Best/average/worst are properties of the inputs; \(O, \Omega, \Theta\) are tools for bounding any of them. “Worst case, \(O(n)\)” answers: no input can cost more than linear.

Why we default to worst case

  • It is a guarantee: no input can be slower.
  • Average case needs an assumption about input distribution — often unknown or wrong.
  • Best case is rarely useful (almost any algorithm can get lucky).
From Unit 3: Dijkstra's algorithm

With a binary-heap priority queue: worst case \(O((|V| + |E|) \log |V|)\).
The bound holds for every graph you can throw at it — that is what makes it a specification you can rely on.

Worked example: early-exit Bubble Sort

Algorithm BubbleSort(L):    swapped ← True    While swapped Do        swapped ← False        Foreach i from 1 To length(L) - 1 Do            If L[i] > L[i+1] Do                swap L[i] and L[i+1]                swapped ← True    Return L
  • Best case (already sorted): one pass, no swaps — \(\Theta(n)\).
  • Worst case (reverse sorted): \(n\) passes of \(n-1\) comparisons — \(\Theta(n^2)\).
  • So: BubbleSort is \(O(n^2)\) and \(\Omega(n)\) — but not \(\Theta(n^2)\) over all inputs.

Analysing iterative pseudocode

The method

  1. Identify the input size \(n\) (list length, \(|V|\) and \(|E|\), digits, …).
  2. Find the loops. For each: how many iterations, as a function of \(n\)?
  3. Nested loops multiply; sequential blocks add.
  4. Watch for loops that don't run \(n\) times: halving (\(\log n\)), triangular (\(n^2/2\)), dependent bounds.
  5. Keep the dominant term; state the tightest bound.

Pattern 1 and 2: single and nested loops

Algorithm MaxOf(L):    best ← L[1]    Foreach i from 2 To n Do        If L[i] > best Do            best ← L[i]    Return best

One loop, \(n-1\) iterations, \(O(1)\) body: \(\Theta(n)\).

Algorithm HasDuplicate(L):    Foreach i from 1 To n Do        Foreach j from 1 To n Do            If i ≠ j And L[i] = L[j] Do                Return True    Return False

\(n \times n\) iterations, \(O(1)\) body: worst case \(\Theta(n^2)\).

Pattern 3: the triangular loop

Algorithm AllPairs(L):    Foreach i from 1 To n Do        Foreach j from i+1 To n Do            process(L[i], L[j])
  • Inner loop runs \(n-1\), then \(n-2\), …, then \(0\) times.
  • Total: \(\dfrac{n(n-1)}{2}\) iterations — still \(\Theta(n^2)\) (constants drop!).
  • Same shape as: comparing every pair of nodes, checking every edge of a complete graph.

Pattern 4: halving loops are logarithmic

Algorithm BinarySearch(L, target):    lo ← 1    hi ← n    While lo ≤ hi Do        mid ← (lo + hi) div 2        If L[mid] = target Do            Return mid        If L[mid] < target Do            lo ← mid + 1        Else Do            hi ← mid - 1    Return not_found
  • The search range halves each pass: \(n \to n/2 \to n/4 \to \dots \to 1\).
  • Number of halvings until 1: \(\log_2 n\). Worst case \(\Theta(\log n)\).
Key idea

Any loop that multiplies or divides its control variable by a constant is logarithmic, not linear.

Putting it together

Algorithm Mystery(L):    total ← 0    Foreach x in L Do              # Block A        total ← total + x    Foreach i from 1 To n Do       # Block B        j ← 1        While j ≤ n Do            total ← total + L[i] * j            j ← j * 2    Return total
  • Block A: \(\Theta(n)\).   Block B: outer \(n\) iterations \(\times\) inner \(\log n\) (doubling) \(= \Theta(n \log n)\).
  • Sequential blocks add: \(\Theta(n) + \Theta(n \log n) = \Theta(n \log n)\).
