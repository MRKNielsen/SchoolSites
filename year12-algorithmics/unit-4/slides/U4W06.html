NP-completeness and reductions

Reductions: solving one problem with another

  • Problem A reduces to problem B if any instance of A can be translated (in polynomial time) into an instance of B whose answer gives A's answer.
  • Consequence: if B is easy, A is easy. Contrapositive: if A is hard, B is at least as hard.
  • Reductions transfer hardness upward — they're how we prove problems hard without solving them.
Example

"Is there a Hamiltonian circuit?" reduces to TSP-decision: set every edge weight to 1, missing edges to 2, ask for a tour of length ≤ \(|V|\). A TSP solver would answer the Hamiltonian question — so TSP is at least as hard as Hamiltonian circuit.

NP-complete: the hardest problems in NP

Definition

A problem is NP-complete if (1) it is in NP, and (2) every problem in NP reduces to it.

  • Cook–Levin (1971): SAT is NP-complete. Karp (1972): 21 more, by reduction chains. Today: thousands, including TSP-decision, graph colouring, knapsack, timetabling…
  • Solve any one of them in polynomial time and you've solved all of NP — that's why nobody believes it's possible, and why P vs NP is worth $1M.
  • Practical reading: if your problem contains an NP-complete core, stop hunting for a fast exact algorithm.

Exact techniques, used well

Dynamic programming: pay once per subproblem

  • Where naive recursion recomputes (Fib's exploding tree), DP stores each subproblem's answer: cost becomes (number of distinct subproblems) × (work each).
  • Fib: \(\Theta(\phi^n) \to \Theta(n)\). Knapsack: \(2^n\) subsets \(\to O(n \cdot W)\) table.
  • Note the fine print: \(O(n \cdot W)\) is polynomial in the value \(W\), not the input length — "pseudo-polynomial". Good enough when capacities are modest.
When DP applies

Overlapping subproblems + optimal substructure (the best whole is built from best parts). If each subproblem is unique, DP buys nothing.

Backtracking: brute force with an early exit

Algorithm Backtrack(partial):    If partial violates a constraint Do        Return                     # prune this whole branch    If partial is complete Do        record solution        Return    Foreach extension of partial Do        Backtrack(partial + extension)
  • Worst case: still exponential. In practice: pruning can eliminate almost the entire tree.
  • Honest reporting: "worst case \(O(2^n)\), but constraint X prunes early in typical instances" — that sentence is exam gold.

Heuristics I: hill climbing

Hill climbing

Algorithm HillClimb(problem):    current ← random solution    While improvement possible Do        best ← best neighbour of current        If best is better than current Do            current ← best        Else Do            Return current    # local optimum
  • Needs: a solution representation, a neighbourhood (small edits), an objective.
  • TSP neighbourhood: swap two cities (2-opt). Each step is cheap and polynomial.
  • Terminates at a local optimum — possibly far from the global one.
The trade

You give up the guarantee of optimality; you get back polynomial time. State both halves whenever you propose a heuristic — that's C9/C10 language.

The local optimum problem

  • Hill climbing stops at the first peak — the objective landscape may hide a taller one across a valley.
  • Cheap fixes: random restarts (climb from many starting points, keep the best), larger neighbourhoods.
  • The principled fix — sometimes accept a worse move — is next week: simulated annealing.
