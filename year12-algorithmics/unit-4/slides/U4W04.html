Divide & conquer, drilled

The divide & conquer template

Algorithm Solve(P):                # problem of size n    If P is small Do        Return direct answer      # base case    split P into a subproblems of size n/b    solve each subproblem recursively    combine the answers            # f(n) work
  • Cost equation: \(T(n) = a\,T(n/b) + f(n)\) — read \(a\), \(b\), \(f\) straight off the code.
  • MergeSort: split in 2, halves, linear merge → \(T(n) = 2T(n/2) + cn = \Theta(n \log n)\) (case 2).
  • Binary search: 1 subproblem, half size, constant check → \(T(n) = T(n/2) + c = \Theta(\log n)\) (case 2).

Criterion 6: consequences

What Criterion 6 actually asks

C6 (from the SAT criteria)

Understanding of the consequences of an algorithm's time complexity on its real-world application.

  • Not: “my algorithm is \(O(|V|^3)\), which is slow.” That's a restatement, not a consequence.
  • A consequence names: the context's realistic input size, what the bound implies at that size (and at growth), and the decision that follows.
  • The bound is maths; the consequence is an argument about your problem.

The consequences paragraph: a template

Four sentences
  1. Bound: “Module X is worst-case \(O(\dots)\) because …” (C5 work, quoted)
  2. Scale: “In this problem's context, the input is realistically \(n \approx \dots\), and could grow to … if …”
  3. Implication: “At that size the algorithm performs roughly … operations, i.e. …; at 10× the size, … times more.”
  4. Decision: “Therefore the design is feasible/marginal/unusable for …, and the bottleneck to address in Unit 4 Outcome 2 is …”
Worked example (facility routing)

“RouteAll is \(\Theta(|V|^3)\) via Floyd–Warshall. A single facility wing has ≈ 40 rooms, so ≈ 64,000 steps — instant. But the multi-wing amendment (A01) grows \(|V|\) to ≈ 400, giving 64 million steps per re-plan; recomputing on every corridor closure would cause visible delay on modest hardware. The all-pairs approach is the bottleneck; a single-source method run on demand is the natural Unit 4 improvement.”

Criterion 7: comparing solutions honestly

  • C7 = comparison of the time complexities of algorithmic solutions to a real-world / applied problem.
  • Compare like with like: same problem, same input measure, both worst-case (or say why not).
  • A comparison has a conclusion: which wins, under what conditions — crossover points matter.
ClaimQuality
“Dijkstra is faster than Floyd–Warshall.”Weak — faster at what, on what graphs?
“For one source, Dijkstra \(O((|V|+|E|)\log|V|)\) beats FW \(\Theta(|V|^3)\); if all pairs are needed and the graph is small and static, FW's simplicity wins.”Strong — conditions and conclusion.

The standard

Every module gets: loop/recurrence structure → tool → tightest bound → one-line justification. Then a whole-solution tally (sequence adds, keep the dominant term) — exactly like the VCAA FriendshipRecommender exemplar.

Workshop method: the module inventory

  1. List every module of your solution (main + helpers). The workbook has the table.
  2. For each: input size variables (\(n\)? \(|V|, |E|\)? both?), structure (loops / recursion / library call), tool, bound.
  3. Library and ADT operations count too: dictionary lookup \(O(1)\), sorting \(O(n \log n)\), your graph ADT's neighbour scan — know its cost.
  4. Tally: compose module bounds along the main control flow; the dominant term is your headline bound.

Memo 02 traps (from marking last year's memos)

  • Hidden costs inside calls: “I call ShortestPath once, so it's \(O(1)\)” — no; the call costs what the callee costs.
  • Wrong input size: analysing in terms of “the list” when the real variable is \(|V|\) and \(|E|\) of your facility graph.
  • Best-case smuggling: quoting the early-exit case as the headline bound. Headline = worst case.
  • Unjustified library costs: if you use sort(), say \(O(n \log n)\) and move on — but say it.
  • No tally: six module bounds and no overall conclusion. The tally paragraph is where C5 marks concentrate.
  • Consequences that aren't: see the consequences template — name sizes, implications, decisions.
