Comparisons that score

The crossover-point method

  • Two candidate solutions rarely have a universal winner — the comparison's job is to find where the lead changes.
  • Example: Dijkstra-per-query vs Floyd–Warshall precompute, for \(q\) routing queries on a static graph:
Set the costs against each other

Per-query Dijkstra: \(q \cdot O((|V|+|E|)\log|V|)\).   FW once, then \(O(1)\) lookups: \(O(|V|^3) + q \cdot O(1)\).

FW wins when \(q\) is large and the graph is small and static; Dijkstra wins when the graph is big, sparse, or changes often. State the condition, then conclude for your context.

Try it!

On your facility (your \(|V|, |E|\)): estimate the \(q\) at which precomputing all pairs pays off.

Anatomy of a full-marks C7 response

  1. Same task, stated: "For the task of ⟨X⟩ on my facility graph…"
  2. Both bounds, justified: quote your C5 work for each candidate.
  3. Conditions: input shape (sparse/dense), query pattern, static vs dynamic.
  4. Crossover: where the lead changes, with arithmetic.
  5. Conclusion for your context: a decision, not a shrug.
Common fail

Comparing different tasks ("Dijkstra finds one path but FW finds all") is description, not comparison. Fix the task first, then compare costs.

AOS2 begins

Tractability

AOS1 asked "how fast is this algorithm?" — AOS2 asks more

  • New question: how fast can any algorithm for this problem be?
  • A problem is tractable if some algorithm solves it in polynomial time — \(O(n^k)\) for some constant \(k\).
  • Sorting: tractable (\(n \log n\)). Shortest path: tractable. But some problems have resisted every attempt at a polynomial algorithm for 70+ years.
Key idea

Complexity is now a property of the problem, not just one algorithm. A clever algorithm can't save you if the problem itself is intractable.

Decision vs optimisation problems

Optimisation formDecision form
TSPFind the shortest tourIs there a tour shorter than \(k\)?
KnapsackMaximise the value packedCan value \(\geq k\) be packed?
Facility routingFind the best sensor placementDo \(m\) sensors suffice?
  • The decision form answers yes/no — this is the form complexity theory classifies.
  • If you can solve the optimisation, you can answer the decision; solving the decision repeatedly (binary search on \(k\)) recovers the optimisation. They stand or fall together.

P and NP

Definitions

P — decision problems solvable in polynomial time.

NP — decision problems where a proposed "yes" answer can be checked in polynomial time.

  • Every problem in P is in NP (if you can solve it fast, you can check it fast).
  • TSP-decision is in NP: given a tour, checking its length is easy — \(O(|V|)\). Finding it seems to require searching \((|V|-1)!\) tours.
  • The million-dollar question: does P = NP? Nobody knows. Almost everyone bets no.

Sudoku: what's the decision problem, and why is it obviously in NP?

Why AOS2 matters for your SAT

  • Memo 03 (due 4 Sep) asks you to improve your solution. If your problem contains an intractable core (many do), "just find the exact optimum faster" is not on the menu.
  • The honest options: solve a restricted version exactly, or the full version approximately — with heuristics. That's the next three weeks.
  • The exam loves this arc: identify intractability → justify a heuristic → analyse the trade-off.
One sentence to remember

Tractable = polynomial; NP = checkable fast; when the problem is (likely) intractable, engineer the trade-off instead of chasing the optimum.
